# loree-web 前端代码约定

> **定位**：在《前端技术架构设计文档》之上，约定 **本仓库** 的目录、命名与数据流习惯，便于多人协作与后续 M2/M3 扩展。  
> **原则**：按 **限界上下文（BC）** 分 feature；页面保持「编排」薄层；可复用逻辑进 `shared/` 或 feature 内 `components/`。

---

## 1. 目录与职责

| 路径 | 职责 |
|------|------|
| `src/app/` | 全局组装：Router、Provider、与路由强相关的壳（不含业务 UI） |
| `src/layouts/` | 跨页布局（如 `AppLayout`） |
| `src/components/ui/` | 设计系统 / shadcn 风格原子组件（无业务语义） |
| `src/shared/` | 横切能力：`api/http`、QueryKey、失效策略、Store、通用反馈、**`shared/form/`** 等 |
| `src/features/<bc>/` | 单个 BC 内聚：`api/`、`schemas.ts`、`components/`、`pages/` |

每个 feature 推荐结构（按需增减文件，**不强行空目录**）：

```text
features/<bc>/
  api/
    types.ts       # 与该 BC 相关的请求/响应类型（与后端 DTO 对齐）
    <bc>-api.ts    # GET/POST/PUT/DELETE 等封装，只做 HTTP + 类型，不写 UI
  schemas.ts       # Zod：表单、内联校验；导出 infer 类型与 toXxxBody 映射函数
  components/      # 仅本 BC 使用的复合组件（工具栏、操作区、表单块等）
  pages/           # 路由级页面：组合 Query + Mutation + 子组件，避免巨型 JSX
```

---

## 2. 命名

| 类型 | 约定 | 示例 |
|------|------|------|
| 页面组件 | PascalCase，默认导出，文件名 `*Page.tsx` | `TaskListPage.tsx` |
| 业务组件 | kebab-case 文件名，命名导出 | `task-list-toolbar.tsx` |
| API 模块 | kebab-case + `-api` 后缀 | `task-api.ts` |
| Zod schema | camelCase + `Schema` 后缀 | `createTaskFormSchema` |
| 映射到请求体 | `toXxxBody` / `toXxxUpsertBody` | `toCreateTaskBody` |
| QueryKey 工厂 | 集中在 `shared/query/query-keys.ts`，按资源命名 | `taskKeys.list(...)` |

---

## 3. 服务端交互

1. **读模型**：`useQuery` + `queryKey` 工厂；列表用 `ListQueryShell` 三态（Skeleton / Empty / Error）。
2. **写模型**：`useMutation`；成功后 **必须** 走 `invalidateAfterCommand(qc, '<command>')`（在 `invalidate-after-command.ts` 登记语义化命令，避免散落字符串）。
3. **HTTP**：统一经 `shared/api/http.ts`；业务代码不直接拼 `fetch` URL 细节（基座与代理由环境/Vite 处理）。

---

## 4. 表单（统一栈）

1. **React Hook Form** + **Zod**：schema 放在对应 feature 的 `schemas.ts`，与页面分离。
2. **Resolver**：一律使用 `shared/form/zod-resolver-typed.ts` 的 **`zodResolverTyped(schema)`**，不在业务文件里重复 `as Resolver<>`。
3. **提交**：`<form onSubmit={handleSubmit(onValid)}>`，成功/取消用 `reset(defaultValues)`；编辑页用 `reset(serverDto)` 灌初始值，并用 ref 等方式避免 **无意义的 refetch 覆盖用户未保存的编辑**（见当前 `TaskEditPage`）。
4. **多表单同页**（如项目列表「新建 + 编辑」）：**各自一个 `useForm`**，互不共享 `watch` 状态，避免串单。

---

## 5. 何时拆组件 / 文件

- 单文件 **明显超过 ~200 行** 或 **同一套字段在两处复用** → 抽 `components/*-form.tsx` 或 `components/*-fields.tsx`，页面只保留数据请求与路由。
- **与路由无关**、可测的纯函数（`toXxxBody`、日期转换）→ 留在 `schemas.ts` 或 `shared` 小 util，不放组件内。

---

## 6. 路由与权限

- 路由表集中在 `src/app/router.tsx`。
- **静态路径优先于动态段**（例如 `tasks/new`、`tasks/:id/edit` 写在 `tasks/:taskId` 之前）。
- 需 **MIND** 的写操作页：用局部 `Mind` 包装（`RoleGuard role="MIND"`），与架构文档角色模型一致。

---

## 7. 导入与类型

- 路径别名 **`@/`** 指向 `src/`。
- 优先 **`import type`** 引入仅类型依赖，减轻循环引用与打包噪声。

---

## 8. 与「实施计划」的关系

- **里程碑 / 勾选**：`docs/前端实施计划.md`（做什么、做到哪）。
- **怎么写代码**：本文档（怎么组织、怎么命名、表单与缓存约定）。

新增 BC 或页面时，先对照本文档补全 `api` + `schemas` + `queryKey`，再写 `pages`，可减少返工。
