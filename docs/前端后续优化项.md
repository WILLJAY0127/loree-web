# loree-web 前端后续优化项

> **定位**：与《前端技术架构设计文档》《前端实施计划》并行维护的 **非阻塞清单**。  
> **原则**：当前迭代以 **先把业务路径跑通**（M1 → M2 → M3 按实施计划）为主；下列项在不影响交付的前提下择机做，避免与主线抢优先级。

---

## 1. UI 与组件拆分

| 项 | 说明 | 触发时机建议 |
|----|------|----------------|
| 任务创建/编辑表单字段复用 | 将 `TaskCreatePage` / `TaskEditPage` 中重复的 label + `register` 区块抽到 `features/task/components/`（如 `task-upsert-fields.tsx` 或对齐架构文档示意名的 `TaskForm`） | 第三次改同一套字段，或 M2 有类似表单要对齐时 |
| 项目列表内联表单 | `ProjectListPage` 若继续变长，可将「新建条」「编辑条」拆为独立小组件 | 单文件明显超过 ~200 行或出现第三套内联表单时 |

---

## 2. 数据层与 hooks

| 项 | 说明 | 触发时机建议 |
|----|------|----------------|
| 任务创建 Mutation | 将 `postTask` + `invalidateAfterCommand` + 成功导航抽成 `useTaskCreateMutation`（或纳入统一 `use-task-mutations`） | 出现第二处「创建任务」入口（如弹窗内创建）时 |
| 任务更新 Mutation | 将 `putTask` + 失效 + Toast 抽成 `useTaskUpdateMutation` | 除编辑页外还有批量编辑等场景时 |
| 编辑页详情 Query | `TaskEditPage` 内联的 `useQuery` 详情可抽为 `useTaskDetailQuery(taskId)`（与 `useTaskDetail` 命名区分：只读 vs 含 Command） | 需要「只读详情」多处复用时 |

---

## 3. 与架构文档 / 约定对齐

| 项 | 说明 |
|----|------|
| 技术架构 S3 示例文件名 | 文档树中为 PascalCase 示例（如 `TaskForm.tsx`）；本仓库业务组件当前多为 kebab-case。二选一统一即可，属风格债。 |
| `features/*/hooks` 覆盖面 | 已覆盖任务列表、任务详情 Command、项目列表 Query；其余页面按需补 hook，不一次性铺全。 |

---

## 4. 业务主线（非「优化」，勿混在本表）

以下条目在 **`前端实施计划.md`** 中勾选推进，**不**作为本文件「可推迟优化」：

- M2：知识点列表/详情、沉淀、标签与推荐、归档、巡检向导等  
- M3：复习列表与作答  
- 阶段 4+：导航体验、错误码、Capacitor 等  

---

## 5. 维护方式

- 讨论新想法时：先判断是 **业务交付** 还是 **结构优化**；后者追加到上表并标建议触发时机。  
- 某优化已落地：在本表对应行标注 **已完成** 或删除该行，避免清单腐烂。
